# AGENTS.md — iKnoWay Personal Site

このファイルは、AI エージェントや開発者がこのプロジェクトを安全に編集するための共通ルールです。
`CLAUDE.md` / `GEMINI.md` / `QWEN.md` はこのファイルへのシンボリックリンクです（直接編集しない）。

## このリポジトリ固有の重要な前提

- **`docs/` は GitHub Pages の公開ディレクトリ（サイト本体）** であり、AI 用ドキュメント置き場ではない。
- AI 用ドキュメントは **`ai-docs/`** に置く（汎用テンプレートで `docs/` と書かれている場所は、このリポジトリでは `ai-docs/` に読み替え済み）。
- コンテンツ（プロフィール・ゲーム・アニメ・映画・成果物・連絡先）の変更は **`docs/shared/data.js` だけを編集する**。各テーマが `window.__data` を読んで描画する。
- バンドラー・フレームワーク不使用。素の HTML / CSS / JS のみ。ビルドステップなし。
- テーマを追加・削除したら `docs/theme-router.js` の `THEMES` 配列と `README.md` を必ず更新する。

## 基本方針

- 既存の設計・命名・ディレクトリ構成を優先する。
- 変更は目的に対して最小限にする。無関係なリファクタリングを混ぜない。
- 読みやすさを優先し、トリッキーな実装や過剰な抽象化を避ける。
- 同じ責務のコード・設定・ドキュメントを複数箇所に分散させない。
- 仕様が曖昧な場合は、推測した前提を明示してから進める。
- 既存のユーザー変更を勝手に戻さない。

## 自動更新ルール

コードや構成を変更した場合、必要に応じて `AGENTS.md`、`DESIGN.md`、`ai-docs/`、`sops/` も同時に更新してください。

更新が必要な例:

- ディレクトリ構成を変えた。
- 主要な機能の置き場所を変えた（テーマ追加・削除を含む）。
- 起動・確認・デプロイ手順を変えた。
- 命名規則やコーディングルールを変えた。
- UI 方針、色、余白、コンポーネント設計を変えた。
- 外部サービス、データ構造（`data.js` のスキーマ）、環境変数の扱いを変えた。
- アーキテクチャ、運用、セキュリティ、意思決定に関わる内容を変えた。

更新しなくてよい例:

- 一時的な実験ファイルを作っただけ。
- コメントや文言の軽微な修正だけ。
- 内部実装を変えたが、利用方法・配置・設計方針が変わらない。

更新するときは、古い情報を残さず現在の実態に合わせてください。

## README の更新ルール

README は外部の人間が初めて見るドキュメントとして書く（`AGENTS.md` / `ai-docs/` は AI・内部向け。重複可だが役割を分ける）。

以下のタイミングで `README.md` も必ず更新する:

- 新しいテーマを追加・削除したとき。
- サイトの構成（URL 構造・ファイル配置）を変えたとき。
- デプロイ方法・外部依存が変化したとき。
- 大きな機能追加・ビジュアル変更があったとき。

## AI ドキュメントの育て方

このリポジトリでは、AI が迷わず作業できるように以下のドキュメントを育てます。
毎回すべて更新する必要はありません。変更の影響があるファイルだけ更新してください。

| ファイル | 役割 | 更新タイミング |
|---|---|---|
| `AGENTS.md` | 最上位の作業ルール | 作業ルール、命名、配置、品質基準が変わったとき |
| `DESIGN.md` | UI / UX / デザイン方針・テーマ実装詳細 | テーマ、スタイル、コンポーネント方針が変わったとき |
| `ai-docs/PROJECT.md` | プロジェクト概要と現在地 | 目的、主要機能、技術スタック、優先事項が変わったとき |
| `ai-docs/CODEMAP.md` | 機能の置き場所 | 主要機能の入口や責務が変わったとき |
| `ai-docs/ARCHITECTURE.md` | 設計と境界 | 構成、データフロー、依存関係が変わったとき |
| `ai-docs/DECISIONS.md` | 意思決定ログ | 重要な設計・技術判断をしたとき |
| `ai-docs/ROADMAP.md` | 優先度と今後の方針 | 開発フェーズ、優先度、やらないことが変わったとき |
| `ai-docs/ENVIRONMENT.md` | 環境と外部サービス | 設定、環境、外部連携が変わったとき |
| `ai-docs/TESTING.md` | 確認コマンドと手動確認 | 確認手順、CI、テスト方針が変わったとき |
| `ai-docs/OPERATIONS.md` | 運用、リリース、復旧 | デプロイ、監視、復旧手順が変わったとき |
| `ai-docs/SECURITY.md` | セキュリティと秘密情報 | 認証、権限、入力境界、秘密情報の扱いが変わったとき |
| `sops/*.sop.md` | 再利用する作業手順 | 同じ作業を次回も使いそうなとき |
| `.cursor/rules/*.mdc` | Cursor 用ルール | Cursor で常に参照したい作業規則が増えたとき |
| `.claude/rules/*.md` | Claude Code 用ルール | 特定のパスを触るときだけ読ませたい規則が増えたとき（`paths:` frontmatter でスコープ可能。常時ルールは `AGENTS.md` に書く） |
| `.github/copilot-instructions.md` | GitHub Copilot 用ルール | Copilot に伝える入口や制約が変わったとき |
| `.claude/skills/` など skill 置き場 | 再利用可能な AI skill | SOP や rules では足りない、横断的な能力が必要なとき |
| `.claude/agents/*.md` | Claude Code 用サブエージェント | 独立したコンテキストで実行させたい定型の役割（レビュー担当、調査担当など）が必要になったとき |
| `.mcp.json` | チーム共有の MCP サーバー設定 | チーム全員に同じ外部ツール連携（MCP）を配りたいとき |

棚卸しの目安:

- 5 回程度の意味ある変更をしたら、`sops/update-ai-docs.sop.md` に従って棚卸しする。
- 30 日程度更新していない場合も、同じ SOP で棚卸しする。
- 大きなリリース、構成変更、外部サービス変更の後は、その場で棚卸しする。

ドキュメントの書き方:

- まだ調べていない・決めていない項目は未記入扱いにする。プレースホルダ文字列は TODO マーカーとして自動検出される。
- このプロジェクトに該当しない項目は「なし」と書く（例: ステージング環境がない場合）。未記入項目を放置しない。

## 鮮度チェックの自動化

棚卸しのタイミングは AI の記憶に頼らず、以下の仕組みで自動検出します。

| 仕組み | 発火タイミング | 内容 |
|---|---|---|
| `scripts/ai-docs-status.sh` | 手動またはフック / CI から | git 履歴からドキュメントの陳腐化と未記入項目の残数を判定する |
| `.claude/settings.json` の SessionStart フック | Claude Code のセッション開始時 | 上記スクリプトの結果を自動でコンテキストに注入する |
| `.claude/commands/update-ai-docs.md` | `/update-ai-docs` 実行時 | 棚卸し SOP を実行する |
| `.github/workflows/ai-docs-freshness.yml` | 週 1 | 陳腐化していたらリマインダーイシューを自動作成する |

AI エージェントへの指示:

- セッション開始時に `[ai-docs]` で始まるメッセージが表示されたら、依頼された作業を優先しつつ、作業の締めくくりで指示された更新（棚卸し、未記入項目の穴埋め）を行ってください。
- 判定基準（コミット 5 件 / 30 日）を変えたい場合は `scripts/ai-docs-status.sh` の先頭の定数を変更してください。

## 追加候補ドキュメントの自動作成

最初から空ファイルを増やさないため、以下は条件を満たしたときに AI が自動作成します。
作成したら、この表の「状態」、`ai-docs/DECISIONS.md`、README のファイル一覧も更新してください。

| ファイル | 作成する条件 | 状態 |
|---|---|---|
| `ai-docs/API.md` | 公開 API・エンドポイント・外部向けインターフェースが増え、`ai-docs/CODEMAP.md` だけでは説明しきれなくなったとき | 未作成 |
| `ai-docs/DATA.md` | `data.js` 以外に主要データ構造が増え、複数箇所から参照されるようになったとき | 未作成 |
| `ai-docs/PERFORMANCE.md` | パフォーマンス要件、計測方法、既知のボトルネックを管理し始めたとき | 未作成 |
| `ai-docs/GLOSSARY.md` | プロジェクト固有の用語が増え、命名や会話で誤解が起きるようになったとき | 未作成 |

作成時のルール:

- 他の ai-docs と同じく、冒頭に「自動更新ルール」（どんな変更のときに更新するか）のセクションを付ける。
- 既存 ai-docs に書かれていた同種の内容は新ファイルへ移し、元の場所には残さない。

## SOP 自動作成ルール

SOP は Standard Operating Procedure の略で、再利用できる作業手順を Markdown で固定するためのドキュメントです。
作業中に「次回も同じ手順で実行しそうな作業」「手順を間違えると影響が大きい作業」「複数ステップの運用・実装・調査作業」が発生した場合は、その作業用の `*.sop.md` を自動作成または更新してください。

作成が必要な例:

- セットアップ、確認、デプロイ、リリース作業。
- 外部サービス連携など、手順ミスの影響が大きい作業。
- データ移行、バックアップ、復旧、障害対応。
- 定期的に繰り返す調査、レビュー、生成、更新作業。
- 新しい種類の機能追加（新テーマ追加など）で、次回の型になる作業。

作成しなくてよい例:

- 1 回限りの軽微な文言修正。
- 既存 SOP に十分な手順がすでに書かれている作業。
- 実験段階で、まだ正式な手順として残す価値がない作業。

配置と命名:

- `sops/` に置く（このリポジトリでは `docs/` はサイト本体なので `docs/sops/` は使わない）。
- ファイル名は `kebab-case` で、内容が分かる `task-name.sop.md` にする。
- 既存 SOP と重複する場合は、新規作成ではなく既存ファイルを更新する。

SOP には最低限、以下を書く:

```text
# 作業名

## Overview
何を、いつ、なぜ実行する手順か。

## Parameters
作業ごとに変わる入力値、対象ファイル、環境、URL など。

## Prerequisites
必要な権限、ツール、事前確認。

## Steps
実行手順。重要な制約は MUST / SHOULD / MAY を使って明確にする。

## Verification
成功確認の方法。

## Rollback / Recovery
失敗時の戻し方、復旧方法、注意点。

## Related Files
関連するコード、設定、ドキュメント。
```

SOP を作成・更新した場合は、作業完了時のまとめに対象ファイル名を含めてください。

## Rules / Skills 自動作成ルール

AI ツール固有の rules や skills は、必要になった場合だけ作成してください。
基本は `AGENTS.md`、`ai-docs/`、`sops/` を優先し、それでも足りない場合に限って rules / skills に切り出します。

判断基準:

- **SOP を作る**: 手順、確認、復旧方法を再利用したい場合。まずこれを選ぶ。
- **rules を作る**: 特定ツールに常時守らせたい短い制約や参照入口が必要な場合。
- **skill を作る**: SOP が AI に参照されず作業ミスが起きた場合（SOP を自動発動させるスタブとして skill 化する）、または複数リポジトリで再利用する専門ワークフロー、追加資料、スクリプト、生成物テンプレートをまとめたい場合。
- **subagent を作る**（Claude Code）: メイン会話とは独立したコンテキストで動かしたい定型の役割（大量のファイルを読むレビュー、調査など）がある場合。`.claude/agents/*.md` に置く。

作成してよい例:

- Cursor だけに常時読ませたい UI 実装ルールを `.cursor/rules/*.mdc` に追加する。
- Claude Code で特定のパスを触るときだけ読ませたい規則（例: `docs/themes/**` のテーマ実装規約）を、`paths:` frontmatter 付きで `.claude/rules/*.md` に追加する。常時読ませたいルールは `AGENTS.md`（CLAUDE.md 経由で毎セッション読み込まれる）に書けば足りるので、rules にしない。
- GitHub Copilot の入口が変わったので `.github/copilot-instructions.md` を更新する。
- AI が `sops/release-or-deploy.sop.md` を見ずにリリースしようとした事故があったので、SOP を参照する skill スタブを `.claude/skills/` に置く。
- 複数プロジェクトで使う専門的なレビュー手順、生成手順、変換手順を skill 化する。

作成しなくてよい例:

- 1 回限りの作業。
- `AGENTS.md` や既存 SOP に書けば十分な内容。
- プロジェクト固有の事実を、ツール別 rules に重複して書くだけの変更。
- まだ運用が固まっていない実験的な手順。

rules / skills を作成・更新した場合は、`ai-docs/CODEMAP.md` または関連 docs から辿れるようにし、必要なら `sops/create-tool-rule-or-skill.sop.md` も更新してください。

| 項目 | 内容 |
|---|---|
| プロジェクト名 | iKnoWay Personal Site |

## ディレクトリ構成

```text
.
├── docs/                  # GitHub Pages 公開ディレクトリ（サイト本体。開発もここで直接行う）
│   ├── index.html         # エントリーポイント（ローダー + テーマ選択）
│   ├── theme-router.js    # テーマ選択ロジック（重み付きランダム、THEMES 配列）
│   ├── themes/<name>/     # 各テーマ（index.html / style.css / script.js）
│   └── shared/            # 全テーマ共通資産（data.js / utils.js / games.js / projects.js / snap.js / human-polish.css）
├── ai-docs/               # AI 用ドキュメント（PROJECT / CODEMAP / ARCHITECTURE など）
├── sops/                  # 再利用する作業手順（*.sop.md）
├── scripts/               # 開発・運用の補助スクリプト（ai-docs-status.sh）
├── .claude/               # Claude Code 設定（settings.json / commands/）
├── .github/               # GitHub Actions（ai-docs 鮮度チェック）
├── AGENTS.md              # AI / 開発者向け作業ルール（各ツール共通の実体。このファイル）
├── CLAUDE.md              # Claude Code 用入口（AGENTS.md へのシンボリックリンク）
├── GEMINI.md              # Gemini CLI 用入口（同上）
├── QWEN.md                # Qwen Code 用入口（同上）
├── DESIGN.md              # UI / UX / デザイン設計・テーマ実装詳細
├── FEATURE_IDEAS.md       # 機能アイデアのメモ
└── README.md              # 外部向けドキュメント
```

## 機能の置き場所

機能を追加・修正するときは、まず `ai-docs/CODEMAP.md` の機能マップで入口を確認してください。
載っていない場合は既存の同種機能（他テーマの実装など）を探し、同じパターンで配置します。
テーマ固有の実装はそのテーマのディレクトリに閉じ、テーマ横断の共通処理だけを `docs/shared/` に置きます。

## プロジェクト固有の開発ルール

- コンテンツの変更は `docs/shared/data.js` のみで行う。テーマ側にコンテンツをハードコードしない。
- 新テーマの読み込み順: `shared/data.js` → `shared/games.js` → `shared/projects.js` → `shared/snap.js` → テーマの `script.js`。CSS は テーマ `style.css` → `shared/human-polish.css` の順。
- ヘッダー・共通操作ボタンの位置・サイズ・ヒットエリアは `docs/shared/human-polish.css` で共通管理する。通常テーマは `site-header` / `nav-inner` / `nav-logo` / `nav-links` / `nav-switch`、特殊テーマは `book-header` / `chapter-nav` / `theme-switch` を使う。寸法変更は各テーマで上書きせず `--chrome-*` 変数を更新する。
- `fantasy-book` のような特殊構造テーマでは、`games.js` / `projects.js` の自動挿入に頼らず、テーマ固有の `script.js` で `data.games` / `data.projects` を独立ページとして描画する。
- テーマ追加手順: `docs/themes/<name>/` を作成 → `window.__data` を読むレンダリングを実装 → `docs/theme-router.js` の `THEMES` に追加 → `README.md` と `ai-docs/CODEMAP.md` を更新。
- `console.log` を本番コードに残さない。
- アニメーションは `transform` / `opacity` のみ使用し、必ず `prefers-reduced-motion` を考慮する（`window.__utils.prefersReducedMotion()`）。
- テーマ間で共通の UI パーツ（ユーティリティ・データ以外）が生まれた場合は `docs/shared/` に切り出す。
- デザインの詳細ルール・各テーマの実装メモ・トレンド方針は `DESIGN.md` を参照する。

## コーディングルール

- 1 つの関数・コンポーネントは 1 つの責務に寄せる。
- ファイルが大きくなりすぎたら、責務単位で分割する。
- 共通化は、同じ処理が実際に複数箇所で必要になってから行う。
- マジックナンバーや固定文字列は、意味が伝わる名前で定数化する。
- エラーは握りつぶさず、呼び出し元が扱える形で返すか、十分な文脈を付けて記録する。
- 外部入力は信用せず、境界で検証する。
- 環境変数・API キー・トークン・個人情報をコミットしない。
- コメントは「なぜそうするか」がコードだけで読み取りにくい場合に書く。
- 依存ライブラリを追加する前に、既存依存・標準 API・小さな実装で足りるか確認する（現在、外部ライブラリ依存はゼロ。維持を優先する）。
- 削除や破壊的変更を行うときは、影響範囲を確認してから実施する。

## 命名規則

| 対象 | 推奨 |
|---|---|
| ディレクトリ | `kebab-case`（テーマ名も。例: `fantasy-book`） |
| 通常ファイル | `kebab-case` |
| 関数 / 変数 | `camelCase` |
| 定数 | `UPPER_SNAKE_CASE` または意味のある `camelCase` |
| CSS クラス | `kebab-case` |
| CSS カスタムプロパティ | `--kebab-case`（例: `--neon-blue`, `--chrome-*`） |

命名は短さよりも意味の明確さを優先します。
名前と実態がずれた場合は、参照箇所も含めて同時に更新してください。

## 開発手順

1. 既存の README、AGENTS、DESIGN、ai-docs、設定ファイルを読む。
2. 変更対象と同種の実装（他テーマなど）を探す。
3. 最小の変更で実装する。
4. `ai-docs/TESTING.md` に従いローカルサーバーで表示確認する。
5. 必要なら `AGENTS.md`、`DESIGN.md`、`ai-docs/`、`sops/` を更新する。
6. 変更内容、確認したこと、未確認のことを簡潔にまとめる。

## 確認コマンド

このプロジェクトに npm / ビルド / lint / 自動テストはありません。確認はローカルサーバーでの手動確認です。

```bash
cd docs && python3 -m http.server 8080
# または
npx serve docs
```

`http://localhost:8080` を開き、`?theme=<id>` で対象テーマを固定して確認します。詳細は `ai-docs/TESTING.md` を参照。

## Git ルール

- 作業前後に `git status` で変更範囲を確認する。
- ユーザーが作った未コミット変更を勝手に戻さない。
- 変更は関係するファイルだけに絞る。
- コミットする場合は、何を変えたかが分かる短いメッセージにする。
- 機密情報や生成物の不要な差分（`.DS_Store` など）を含めない。
- `main` へのプッシュはそのまま本番公開（GitHub Pages）になることを意識する。

## やらないこと

- 目的と関係ない大規模リファクタリング。
- 使っていないライブラリや設定の追加（外部依存ゼロを維持）。
- 一時ファイル、ログ、ビルド成果物のコミット。
- 既存の設計を確認しないまま別パターンを持ち込むこと。
- ドキュメントと実装が食い違った状態で完了すること。
- バックエンド・AI 駆動 UI・音声・AR / VR などスコープ外機能の追加（`ai-docs/ROADMAP.md` の Won't Do を参照）。
